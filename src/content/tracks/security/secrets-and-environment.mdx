A leaked secret — API key, DB password, signing key — can compromise everything
behind it. In Next.js the most common leak is accidentally shipping a secret to
the browser.

## The NEXT*PUBLIC* rule

Any environment variable prefixed `NEXT_PUBLIC_` is **inlined into the client
bundle** and visible to everyone. Everything else stays server-only.

```ts
// server-only — safe
const key = process.env.STRIPE_SECRET_KEY;

// ❌ NEVER do this — your secret is now in every user's browser
const key = process.env.NEXT_PUBLIC_STRIPE_SECRET_KEY;
```

<Callout type="danger" title="The prefix is a publishing decision">
  Adding `NEXT_PUBLIC_` to a secret publishes it. Only public, non-sensitive
  values (analytics IDs, public API base URLs) deserve the prefix. When unsure,
  leave it off.
</Callout>

## Keep secret reads on the server

Read secrets only in Server Components, Server Actions, and route handlers.
Guard server-only modules so a client import fails the build:

```ts
import "server-only"; // throws if this module is imported into client code
const apiKey = process.env.PAYMENTS_KEY;
```

## Don't commit secrets

```bash
# .gitignore
.env*.local
.env
```

- Commit a `.env.example` with **keys but no values** as documentation.
- If a secret ever lands in git, **rotate it** — deleting the commit isn't enough;
  it's in the history and likely already scraped.
- Add a scanner (gitleaks, trufflehog, GitHub secret scanning) to CI to catch
  leaks before they merge.

## Manage secrets properly in production

Use a secrets manager or your platform's encrypted env store (Vault, AWS/GCP
Secrets Manager, Doppler) rather than plaintext files on disk. Scope each secret
to least privilege and rotate regularly.

<Callout type="info" title="Validate env at startup">
  Parse `process.env` through a schema (Zod) when the app boots so a missing or
  malformed secret fails loudly at deploy time, not at 3am in production.
</Callout>
