To reason about performance you need a precise mental model of what React does
each time state changes.

## Render is not "update the DOM"

A **render** is React calling your components to produce a tree of elements
(the description). **Commit** is the separate phase where React applies the
minimal set of changes to the real DOM. A component can render many times
without any DOM change.

```text
state change → render (build element tree) → reconcile (diff) → commit (DOM)
```

## Reconciliation: the diff

React compares the new element tree with the previous one:

- **Same type at the same position** → reuse the DOM node, update changed props.
- **Different type** → unmount the old subtree, mount the new one (state is lost).
- **Lists** → matched by `key`; stable keys let React move instead of rebuild.

<Callout type="warning" title="Same component, different position = remount">
  Rendering a component conditionally in two different branches can remount it
  and reset its state. Keep components in a stable position, or give them a
  stable `key`, to preserve state.
</Callout>

## What triggers a re-render

A component re-renders when:

1. Its **state** changes (`setState`).
2. Its **parent** re-renders (by default, children re-render too).
3. A **context** it consumes changes.

Note: a parent re-rendering re-renders children even if their props are
identical — unless they're wrapped in `React.memo`.

## Controlling re-renders

- **`React.memo`** to skip children whose props didn't change.
- **Move state down** so fewer components sit above it.
- **Pass JSX as `children`** — children passed from above don't re-render when
  the wrapper's own state changes.

```tsx
// <Wrapper> can update its own state without re-rendering <ExpensiveTree/>
<Wrapper>
  <ExpensiveTree />
</Wrapper>
```

<Callout type="info" title="Profile before you optimize">
  Use the React DevTools Profiler's "highlight updates" to see what actually
  re-renders. Most "slow" React is a handful of components rendering far too
  often — fix those, don't memoize blindly.
</Callout>
